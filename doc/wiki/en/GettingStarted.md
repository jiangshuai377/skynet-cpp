# GettingStarted
## Current Implementation Status

The current runtime uses the preload bootstrap path: set `SKYNET_THREAD` for worker count and `SKYNET_PRELOAD` for the preload script. The preload script configures Lua path/cpath/service path, starts launcher, and selects the application entry. Test entrypoints are split into `tests/logic`, `tests/stress`, and `tests/perf`, with separate coverage and Linux Docker perf runners. Actor scheduling now uses `ActorQueue`, sharded registry, and atomic wakeup; Lua callback and `skynet.core` actor context are cached on the hot path.

## Framework

skynet-cpp is a lightweight Actor-model server framework. You can think of it as a simple operating system that schedules thousands of Lua virtual machines to work in parallel. Each Lua VM can receive and process messages sent by other VMs, and send messages to them.

skynet-cpp has built-in management for external network data input and timers, converting them into consistent message inputs for each service.

### Relationship with Original skynet

skynet-cpp's design philosophy and API semantics are entirely derived from [cloudwu/skynet](https://github.com/cloudwu/skynet), but the underlying framework is reimplemented in C++20. For Lua developers, API usage is largely consistent with the original skynet.

## Service

skynet-cpp services are written in Lua. You simply place a conforming `.lua` file in a path accessible to skynet-cpp, and it can be launched by other services. Each service has a unique 32-bit address (handle) assigned by the framework.

Each service goes through three phases:

1. **Load phase**: The service source file is loaded and executed. Blocking APIs **must not** be called during this phase.
2. **Init phase**: The initialization function registered via `skynet.start(func)` is executed. Any skynet API can be called during this phase. The `skynet.newservice` call that launched this service will wait until initialization completes.
3. **Work phase**: After initialization completes, services with registered message handlers begin responding to messages.

```lua
local skynet = require "skynet"

-- Load phase: set module-level variables
local CMD = {}

function CMD.hello(...)
    return "world"
end

skynet.start(function()
    -- Init phase: register message dispatch
    skynet.dispatch("lua", function(session, source, cmd, ...)
        local f = assert(CMD[cmd])
        skynet.retpack(f(...))
    end)
end)
```

## Message

Each skynet-cpp message consists of the following elements:

1. **session**: A unique identifier generated by the requesting service. The responder includes the session in its reply, allowing the sender to match requests with responses. A session of 0 means no response is needed (one-way push).
2. **source**: The address (32-bit handle) of the service that sent the message.
3. **type**: The message category. The most commonly used is `"lua"`, for communication between Lua services.
4. **message + size**: The message content (C pointer + length), generated by a serialization function.

### Message Types

| Type | Name | Purpose |
|---|---|---|
| 0 | `text` | Plain text message |
| 1 | `response` | RPC response |
| 6 | `socket` | Network event |
| 7 | `error` | Error notification |
| 10 | `lua` | Lua serialized message (most common) |

## Coroutine Scheduling

At the low level, each service is simply a message processor. But at the application level, it leverages Lua's coroutine mechanism.

When your service sends a request to another service (`skynet.call`), the current coroutine is suspended. Once the other service receives the request and sends a response, the framework locates the suspended coroutine, passes the response data in, and resumes the previously interrupted business flow. From the user's perspective, it feels more like an independent thread handling the business logic.

**Reentry warning**: While a service's business flow is suspended, it can still process other messages. Therefore, the internal state of a service obtained before a `skynet.call` may have already changed by the time it returns. The execution between two blocking API calls is atomic. You can use [CriticalSection](CriticalSection.md) to reduce the complexity caused by pseudo-concurrency.

## Network

skynet-cpp has a built-in network layer that wraps TCP and UDP functionality. It is not recommended to use any modules that directly interact with system network APIs in your services, because once blocked by network IO, it affects the entire worker thread.

Using skynet-cpp's built-in [Socket](Socket.md) API allows the CPU to be fully utilized even when network IO is blocking.

It is recommended to use [GateServer](GateServer.md) gateway service for managing client connections.

## External Services

skynet-cpp provides driver modules for [Redis](ExternalService.md#redis-驱动), [MySQL](ExternalService.md#mysql-驱动), and [MongoDB](ExternalService.md#mongodb-驱动). These driver modules are all built on [SocketChannel](SocketChannel.md) and integrate well with skynet-cpp.

## Cluster

skynet-cpp implements cluster mode for cross-node RPC. See [Cluster](Cluster.md) for details.

Unlike the original skynet, skynet-cpp **does not support** master/slave mode (harbor mode). It is recommended to use cluster mode exclusively.

## Differences from Original skynet

- **Does not support** master/slave (harbor) mode
- **Does not support** the Snax framework
- **Does not support** the Sproto protocol
- **Does not support** DataCenter (deprecated)
- ShareData uses message passing deep copy instead of C shared memory
- Uses Lua 5.5.0 (original uses Lua 5.4)
- Database drivers (BSON/SHA1) are all pure Lua implementations
